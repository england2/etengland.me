## Intro

webterm is a microservice-style application that provides a web interface to access Linux containers hosted in a Kubernetes cluster.

Technologies involved are Go, Kubernetes, ArgoCD, Docker, Helm, Node, and Javascript.

### Technical Overview

- Webterm has three deployable services: `webserver`, `pseudo-terminal-manager`, and `pseudo-terminal`
- We use a Go program for cluster orchestration and terminal allocation, and Node.js as a terminal backend system
- Website hosts a static bundle serving a terminal UI based on `xterm.js`
- Deployed using a Helm Chart, with image tags updated by GitHub Actions in [`pipeline.yml`](https://github.com/england2/webterm/blob/main/.github/workflows/pipeline.yml)
  - (see [shellbin](https://github.com/england2/shellbin) for an explanation of a similar pipeline.)
- Terminal pods use `node-pty` to bind a real shell process to a browser socket connection; keyboard inputs and results are send over the wire

###  Source code

The source for webterm is available [here](https://github.com/england2/webterm).

## Microservices Overview

- `webserver`
  - serves the static frontend bundle (e.g index.js, index.html)
  - exposes the `/getPseudoTerminalAddress` API used by the browser
  - rewrites the returned terminal address into something the browser can dial

- `pseudo-terminal`
  - represents a user-facing terminal pod
  - runs `node-pty` within Node.js to commicate terminal data with the frontend's `xterm.js`
  - forwards browser input to the shell and streams shell output back to the browser
  - asks the manager to kill the pod after idle timeout

- `pseudo-terminal-manager`
  - talks to the Kubernetes API from inside the cluster
  - tracks which terminal pods are free, in use, or are being recreated
  - automatically exposes pseudoterminals with a nodePort; provides address to pseudoterminals to the browser
  - creates additional pseudoterminals by scaling a StatefulSet

## General Connection Flow

The backend needs to provide each user with a pseudoterminal backend.

Here's a rough overview of this process:
- the browser loads the frontend from `webserver`
- frontend calls `webserver` for a terminal address
- `webserver` forwards the request to `pseudo-terminal-manager`
- manager first checks whether that client already owns a pod
- if yes, the existing pod is returned and the session is treated as a reconnect
- if no, the manager allocates a spare pod in state `ready first`
- if the last spare pod is in used, the `pseudo-terminal-manager` scales the StatefulSet up so a new spare is created
- the browser receives the address of a `pseudo-terminal`
- the browser then opens a socket connection directly to the assigned terminal pod

## What Are Pseudoterminals For?

We're making a web terminal, but unfortunately the browser cannot run Linux or host a real shell process by itself.

The browser can only render a terminal-like interface and send user input over the network. The actual shell has to run somewhere else, in this case inside a container in the cluster.

This process is enabled by pseudoterminals.

Basically, a pseudoterminal is an OS primitive that attaches to a running process. They are important, as all modern terminal emulators (even local) use pseudoterminals as an abstraction. It can be thought of that pseudoterminals "exchange" keyboard input for terminal process output. 

For a web-based terminal, instead of handling input and output locally, the browser sends user input over the network to a backend container running a pseudoterminal, and the resulting output is streamed back to the browser.

Here’s a quick rundown of how webterm uses pseudoterminals:
- xterm.js handles the terminal UI in the browser, but it is not running the shell
- When a user connects, the frontend gets assigned a specific terminal pod and opens a socket connection to it
- Inside that pod, node-pty creates a pseudoterminal and starts the real shell process
- Keystrokes from the browser are sent over the socket into that pseudoterminal
- The pseudoterminal sends output through the socket which is rendered by xterm.js

See [containerPseudoTerminal.js](https://github.com/england2/webterm/blob/master/pseudo-terminal/to-copy-inside/containerPseudoTerminal.js) for the actual pseuodoterminal server running in terminal client containers.

## Filtering the Kubernetes API to Manage Pods

Kubernetes exposes a watch mechanism, which is basically a way to subscribe to changes in the cluster objects constantly polling the API.

`pseudo-terminal-manager` uses this to notice when a terminal pod changes state, especially when a pod is being recreated or when a newly scaled pod becomes ready.

Here's a some context for the code below:
- The manager instanties a Kubernetes watch on pod objects in its namespace, so it receives a stream of events whenever those pods change state or are updated
- That stream contains updates about many pods at once
- In practice, the manager is often only interested in certain events for one specific pod
- Specifically, we'll looking for a _pattern of events_ that indicates the pod is ready
- The purpose of the filter is to take the shared watch stream and extract only the events related to the pod we care about
- Using concurrency, the filter can keep running while new pod-specific conditions are added or removed
- This lets the manager watch one stream of Kubernetes events while still thinking about individual pods separately


[filter.go](https://github.com/england2/webterm/blob/main/pseudo-terminal-manager/pseudo-terminal-manager-api/filter.go)

```go
for {
    select {
    case paramToAppend := <-fil.paramStream:
        fil.params = append(fil.params, &paramToAppend)
    case indexToRemove := <-fil.remIndexChan:
        fil.params = remove(fil.params, indexToRemove)
    case event := <-fil.inChan:
        for _, fp := range fil.params {
            if fp.pass(event, fil.done) {
                fp.outChan <- event
            }
        }
    case <-fil.done:
        return
    default:
        if len(fil.params) == 0 {
            close(fil.done)
            runningFilter = nil
        }
    }
}
```
This code works well enough for this project, but in the future I will probably use [kubebuilder](https://github.com/kubernetes-sigs/kubebuilder) for interacting with the Kubernetes API!
